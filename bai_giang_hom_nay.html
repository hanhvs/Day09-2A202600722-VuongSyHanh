🎯 Tổng quan bài học

Hôm nay bạn sẽ xây dựng một hệ thống tư vấn pháp lý đa tác nhân (Legal Multi-Agent System), đi qua 5 giai đoạn tiến hóa của LLM: từ gọi API đơn giản → tích hợp tools/RAG → agent tự động → nhiều agents song song → và cuối cùng là hệ thống phân tán giao tiếp qua HTTP với protocol A2A.

Mục tiêu học tập

  • Hiểu cách LLM hoạt động từ cơ bản đến nâng cao
  • Biết cách tích hợp tools và RAG (Retrieval-Augmented Generation) vào LLM
  • Xây dựng single agent với pattern ReAct (Reasoning + Acting)
  • Tạo multi-agent system bằng LangGraph StateGraph + Send API
  • Triển khai distributed agents giao tiếp qua A2A Protocol với dynamic discovery

Bài toán: Hệ thống tư vấn pháp lý

Người dùng đặt câu hỏi pháp lý → hệ thống phân tích đồng thời 3 góc nhìn: Luật chung, Thuế, và Compliance (SEC/SOX/GDPR) → tổng hợp thành câu trả lời toàn diện.

🏗️ Kiến trúc hệ thống (Stage 5 — đích đến)

┌─────────────────────┐ │ Registry Service │ :10000 │ /register │ │ /discover/{task} │ └─────────┬───────────┘ │ (agents tự đăng ký khi khởi động) ┌────────────────────┼─────────────────────┐ │ │ │ Tax Agent :10102 Law Agent :10101 Compliance Agent :10103 │ │ │ └─────────► delegate song song ◄───────────┘ │ Customer Agent :10100 │ User

Vai trò từng Agent

AgentPortPatternNhiệm vụ

Request Flow

User question → Customer Agent: LLM phát hiện domain pháp lý, gọi tool delegate → Registry: discover("legal_question") → trả endpoint Law Agent → Law Agent: [analyze_law] LLM phân tích contract/tort [check_routing] LLM quyết định: cần Tax? cần Compliance? [call_tax] ──→ Registry → Tax Agent (A2A) ┐ [call_compliance] ──→ Registry → Compliance (A2A) ├ song song [aggregate] Tổng hợp các phân tích ┘ → Customer Agent trả response cho user

Key Design Patterns

  • Dynamic discovery — agents tìm nhau qua Registry, không hardcode URL
  • Parallel delegation — LangGraph Send API dispatch song song
  • Trace propagation — trace_id/context_id đi xuyên qua mỗi A2A hop
  • Depth guards — MAX_DELEGATION_DEPTH = 3 chống vòng lặp vô hạn
  • Annotated reducers — Annotated[str, _last_wins] xử lý ghi song song vào state

🔵 Stage 2 — LLM + RAG & Tools ⭐⭐ Trung bình

RAG (Retrieval-Augmented Generation) cho phép LLM tra cứu knowledge base trước khi trả lời. Tools là các function LLM có thể gọi để tính toán hoặc truy vấn data.

Function Calling Flow

1. LLM nhận câu hỏi + tools
→
2. LLM chọn tool
→
3. Tool execute
→
4. LLM tổng hợp

Bài tập 3.1 — Thêm tool tra cứu án lệ

@tool
def search_case_law(keywords: str) -> str:
    """Tìm kiếm án lệ theo từ khóa."""
    cases = {
        "breach": "Hadley v. Baxendale (1854) - Consequential damages",
        "negligence": "Donoghue v. Stevenson (1932) - Duty of care",
        "contract": "Carlill v. Carbolic Smoke Ball Co (1893)",
    }
    for key, case in cases.items():
        if key in keywords.lower():
            return case
    return "Không tìm thấy án lệ phù hợp"

🟠 Stage 4 — Multi-Agent In-Process ⭐⭐⭐⭐ Khó

Nhiều agents chuyên môn hóa cùng làm việc trong cùng một process. Mỗi agent tập trung vào domain riêng và có thể chạy song song.

Các thành phần LangGraph

  • StateGraph — định nghĩa graph với state chia sẻ
  • TypedDict — schema của shared state
  • Send() API — dispatch nhiều tasks song song
  • add_node() / add_edge() — xây dựng luồng

Cấu trúc graph (Stage 4)

┌──────────────┐ │ analyze_law │ └──────┬───────┘ │ ┌──────▼───────┐ │ check_routing│ ← LLM quyết định └──────┬───────┘ Send() │ Send() ┌────────┴────────┐ ▼ ▼ ┌─────────────┐ ┌────────────────┐ │ tax_agent │ │compliance_agent│ ← chạy song song └──────┬──────┘ └────────┬───────┘ └─────────┬─────────┘ ▼ ┌────────────────┐ │aggregate_results│ └────────────────┘

Bài tập 4.1 — Thêm privacy_agent

def privacy_agent(state: State) -> dict:
    """Agent chuyên về GDPR và luật bảo vệ dữ liệu cá nhân."""
    llm = get_llm()
    prompt = f"""Bạn là chuyên gia về GDPR và privacy.
Câu hỏi gốc: {state['question']}
Phân tích pháp lý: {state.get('law_analysis', 'N/A')}
Hãy phân tích các vấn đề về privacy và GDPR (nếu có).
"""
    response = llm.invoke([HumanMessage(content=prompt)])
    return {"privacy_analysis": response.content}

Bài tập 4.2 — Conditional routing

Chỉ gọi privacy_agent khi câu hỏi có từ khóa "data", "privacy", "gdpr", "dữ liệu":

def check_routing(state: State) -> list[Send]:
    q = state["question"].lower()
    tasks = []
    if any(k in q for k in ["tax", "irs", "thuế"]):
        tasks.append(Send("tax_agent", state))
    if any(k in q for k in ["compliance", "sec", "regulation"]):
        tasks.append(Send("compliance_agent", state))
    if any(k in q for k in ["data", "privacy", "gdpr", "dữ liệu"]):
        tasks.append(Send("privacy_agent", state))
    return tasks if tasks else [Send("aggregate_results", state)]

🔴 Stage 5 — Distributed A2A System ⭐⭐⭐⭐⭐ Production

A2A (Agent-to-Agent) Protocol: chuẩn giao tiếp giữa các agents qua HTTP. Mỗi agent là một service độc lập, có thể scale riêng biệt.

Khác biệt Stage 4 vs Stage 5

Khía cạnhStage 4 (In-Process)Stage 5 (Distributed)
Giao tiếpFunction callHTTP + A2A protocol
Triển khai1 process5 services riêng biệt
DiscoveryImport trực tiếpDynamic qua Registry
ScalingToàn bộ systemTừng agent riêng
Fault toleranceKhôngMột agent fail không sập hệ thống

Cấu trúc mỗi agent module

  • graph.py — định nghĩa LangGraph graph
  • agent_executor.py — bridge giữa A2A SDK và LangGraph
  • __main__.py — server bootstrap, agent card, đăng ký với Registry

🌟 Bài tập cộng điểm

Sau khi chạy đầy đủ Stage 5 (test_client.py), trả lời 2 câu hỏi:

  1. Latency (tổng thời gian trả lời 1 câu hỏi của hệ thống) là bao nhiêu giây?
  2. Đề xuất phương án giảm latency và demo + show thời gian đã giảm được sau khi áp dụng.

Gợi ý các hướng tối ưu latency

  • Parallel hóa nhiều hơn — kiểm tra xem có nhánh nào có thể chạy đồng thời mà đang chạy tuần tự không
  • Streaming response — trả lời từng phần thay vì chờ toàn bộ
  • Caching — cache câu hỏi tương tự ở Registry hoặc agent level
  • Đổi model nhanh hơn — Sonnet → Haiku cho các agent đơn giản
  • Giảm system prompt — prompt càng ngắn càng nhanh
  • Connection pooling — reuse HTTP connections giữa các A2A calls
  • Bỏ delegation depth không cần thiết — flatten architecture khi có thể
