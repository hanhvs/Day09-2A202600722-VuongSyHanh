Stage 4 — Multi-Agent In-Process

Nhiều specialist agents cộng tác qua LangGraph StateGraph + Send API

⭐⭐⭐⭐ Khó 30 phút StateGraph · Send() · Annotated reducers

1. Ý tưởng

Stage 3 dùng 1 agent ôm hết (law + tax + compliance trong 1 system prompt). Stage 4 tách thành nhiều specialists, mỗi agent:

  • Có system prompt riêng tập trung 1 domain
  • Có tool riêng (vd tax có search_tax_law, compliance có search_compliance_law)
  • Chạy song song qua LangGraph Send() API
💡 Tại sao tách? Cùng câu hỏi nhưng 3 góc nhìn khác nhau (luật / thuế / compliance) — nếu nhồi vào 1 prompt thì LLM dễ trộn lẫn, mỗi domain bị nông. Tách ra → mỗi agent có "không gian" riêng để phân tích sâu.

3. 4 thành phần LangGraph quan trọng

3.1 — StateGraph + TypedDict

State là dữ liệu chia sẻ giữa các nodes. Mỗi node nhận state làm input, return một dict update vào state.

class LegalState(TypedDict):
    question: str
    law_analysis: str
    needs_tax: bool
    needs_compliance: bool
    tax_result: Annotated[str, _last_wins]      # ← reducer!
    compliance_result: Annotated[str, _last_wins]
    final_answer: str

3.2 — Annotated[..., reducer] reducer

Khi nhiều nodes ghi vào cùng 1 field song song, LangGraph cần biết cách merge. Annotated chỉ định reducer:

def _last_wins(a: str, b: str) -> str:
    return b if b else a

tax_result: Annotated[str, _last_wins]
⚠️ Nếu quên Annotated mà có 2 nodes cùng ghi → LangGraph báo "INVALID_CONCURRENT_GRAPH_UPDATE". Mỗi field có 2+ writer song song bắt buộc phải có reducer.

3.3 — Send() API (parallel dispatch)

Send(node_name, state) dispatch task tới node — nếu trả về list[Send] trong conditional edge, các tasks chạy song song:

def route_to_specialists(state) -> list[Send]:
    sends = []
    if state["needs_tax"]:
        sends.append(Send("call_tax_specialist", state))
    if state["needs_compliance"]:
        sends.append(Send("call_compliance_specialist", state))
    return sends

3.4 — add_conditional_edges

graph.add_conditional_edges(
    "check_routing",                                    # ← từ node này
    route_to_specialists,                               # ← hàm quyết định
    ["call_tax_specialist", "call_compliance_specialist", "aggregate"],   # ← các đích
)

💡 Trade-off của LLM-based routing:
  • ✅ Hiểu được ngữ cảnh phức tạp + đa ngôn ngữ (VN/EN trộn lẫn)
  • ❌ Không deterministic — có thể sai (so với keyword match cứng)
  • 💰 Cost: thêm 1 LLM call cho mỗi câu hỏi, nhưng tiết kiệm được N-1 specialist calls không cần thiết

6. Lưu ý kỹ thuật quan trọng

Tại sao cần reducer cho tax_result nhưng không cần cho law_analysis?

Vì law_analysis chỉ ghi 1 lần ở node analyze_law (không có node nào khác ghi song song). Còn tax_result, compliance_result, privacy_result — nếu Send() dispatch nhiều specialist, mỗi specialist trả về dict có thể trùng key → cần reducer.

Khi nào dùng conditional_edges vs add_edge?

  • add_edge(A, B) — luôn đi từ A → B
  • add_conditional_edges(A, fn, [...]) — đi đâu phụ thuộc fn(state) trả về

Specialist trong Stage 4 chính là ReAct agent (Stage 3)

Mỗi node specialist nội bộ tạo ra một create_react_agent. Tức là multi-agent = các single agents được orchestrate bởi graph.
