Stage 5 — Distributed A2A System

Mỗi agent là 1 service độc lập, giao tiếp qua HTTP + A2A protocol, dynamic discovery

⭐⭐⭐⭐⭐ Production 30 phút A2A SDK · FastAPI · Registry

1. Ý tưởng — Tại sao cần Stage 5?

Stage 4 đã có topology multi-agent đúng, nhưng tất cả chạy chung 1 process. Production-ready cần:

  • Scale từng agent riêng — Tax Agent bị nghẽn? Spin thêm 5 replicas, không động đến Compliance
  • Fault tolerance — Compliance Agent crash, hệ thống vẫn trả được Tax + Law
  • Polyglot — Tax Agent có thể viết bằng Python, Compliance bằng Go, miễn cùng A2A protocol
  • Owned by different teams — Team A deploy Tax Agent, team B deploy Compliance, không cần share codebase
💡 Key insight: Stage 5 giữ nguyên graph topology của Stage 4. Chỉ đổi agent.ainvoke() in-process thành delegate(endpoint, ...) qua HTTP. Hiểu Stage 4 vững → Stage 5 là extension tự nhiên.

3. A2A Protocol — Khái niệm cốt lõi

A2A (Agent-to-Agent) là chuẩn của Google cho agents giao tiếp qua HTTP. Có 4 khái niệm cần nhớ:

Khái niệmÝ nghĩaTương tự như
AgentCard Metadata mô tả agent: tên, version, skills, URL. Public ở /.well-known/agent.json OpenAPI spec / package.json
Task Một "đơn vị công việc" mà agent xử lý. Có ID, state, lifecycle Job trong queue
Message Trao đổi giữa client ↔ agent. Có role, parts, metadata HTTP request/response body
Part Nội dung trong message. Có thể là TextPart, FilePart, DataPart multipart/form-data

Agent Card mẫu (Tax Agent)

AgentCard(
    name="Tax Agent",
    description="Specialist tax attorney and CPA",
    url="http://localhost:10102",
    version="1.0.0",
    capabilities=AgentCapabilities(streaming=False),
    skills=[
        AgentSkill(
            id="tax_question",                          # ← key để discover
            name="Tax Question",
            description="Answer questions about tax law...",
            tags=["tax", "irs", "penalties"],
        )
    ],
)

Lifecycle 1 request

  1. Client fetch {endpoint}/.well-known/agent.json để verify agent
  2. Client gọi send_message(request) với SendMessageRequest
  3. Server tạo Task, chuyển trạng thái: submitted → working → completed
  4. Server trả Task có artifacts (kết quả) hoặc Message trực tiếp
  5. Client parse text từ artifacts/parts

4. Service Discovery — không hardcode URL

Vấn đề: Stage 4 hardcode call_tax import từ tax_agent. Stage 5 không thể vì 2 services chạy độc lập.

Flow đăng ký + tìm nhau

Startup — Tax Agent đăng ký: 
POST http://localhost:10000/register
{
    "agent_name": "tax-agent",
    "endpoint": "http://localhost:10102",
    "tasks": ["tax_question"],
    "tags": ["tax", "irs", "penalties"]
}
Runtime — Law Agent tìm Tax Agent: 
GET http://localhost:10000/discover/tax_question
→ {"endpoint": "http://localhost:10102", "agent_name": "tax-agent"}
Delegate qua A2A: 
endpoint = await discover("tax_question")           # registry call
result   = await delegate(endpoint, question, ...)  # A2A HTTP call
💡 Ưu điểm dynamic discovery:
  • Đổi port Tax Agent từ 10102 → 10200? Chỉ sửa __main__.py, mọi caller tự cập nhật
  • Cần 2 Tax Agent (load balance)? Registry trả endpoint khác nhau theo round-robin
  • Tax Agent down? Registry trả 404, Law Agent tự handle exception

Retry registration

Khi start_all.sh chạy, Registry start trước. Agents khác có thể start trước khi Registry sẵn sàng → mỗi agent có hàm _register_with_retry() retry 10 lần × 2 giây:

for attempt in range(1, max_attempts + 1):
    try:
        await register(info)
        return
    except Exception:
        await asyncio.sleep(delay)

5. Trace Propagation — debug xuyên service

Khi 1 request đi qua 4 services, làm sao biết log nào thuộc request nào? Dùng trace_id:

  1. Customer Agent generate trace_id = uuid4() khi nhận request đầu tiên
  2. Mỗi A2A call gắn trace_id vào message.metadata
  3. Mọi log statement trong mọi agent đều prefix trace_id
  4. Khi debug: grep trace=<id> ở 4 file log → ra chuỗi đầy đủ
message = Message(
    role=Role.user,
    parts=[Part(root=TextPart(text=question))],
    message_id=str(uuid4()),
    context_id=context_id,
    metadata={
        "trace_id": trace_id,            # ← propagate
        "context_id": context_id,
        "delegation_depth": depth,       # ← cũng propagate
    },
)

6. Depth Guard — chống infinite delegation

Multi-agent có rủi ro: Law Agent gọi Tax Agent → Tax Agent (bug) lại gọi Law Agent → infinite loop.

Cơ chế bảo vệ trong law_agent/graph.py:

MAX_DELEGATION_DEPTH = 3

async def check_routing(state: LawState) -> dict:
    depth = state.get("delegation_depth", 0)
    if depth >= MAX_DELEGATION_DEPTH:
        logger.info("Max delegation depth reached (%d); skip sub-agents", depth)
        return {"needs_tax": False, "needs_compliance": False}
    # ... bình thường

# Mỗi lần delegate, depth + 1
await delegate(
    endpoint=endpoint,
    question=...,
    depth=state.get("delegation_depth", 0) + 1,    # ← increment
)
⚠️ Pattern phổ biến trong distributed agents: luôn cap depth. Tương tự DNS có TTL, HTTP có max-redirects.

7. Cấu trúc 1 agent module

Mỗi agent (tax/compliance/law/customer) có cùng 3 file:

FileVai trò
graph.py Logic agent — LangGraph graph (giống Stage 4)
agent_executor.py Bridge giữa A2A SDK và LangGraph — nhận Task từ A2A, gọi graph, trả Artifact
__main__.py Server bootstrap — tạo AgentCard, đăng ký Registry, start uvicorn

Pattern bootstrap chung:

# Bước 1: đăng ký với Registry (retry nếu Registry chưa up)
await _register_with_retry()

# Bước 2: tạo AgentCard
agent_card = AgentCard(name="Tax Agent", skills=[...])

# Bước 3: dùng A2A SDK build FastAPI app
executor = TaxAgentExecutor()
request_handler = DefaultRequestHandler(agent_executor=executor, ...)
app = A2AFastAPIApplication(agent_card=agent_card, http_handler=request_handler).build()

# Bước 4: chạy uvicorn
uvicorn.run(app, host="0.0.0.0", port=PORT)

10. Bài tập cộng điểm (Latency optimization)

Sau khi chạy đầy đủ Stage 5, trả lời 2 câu hỏi:

  1. Latency (tổng thời gian trả lời 1 câu hỏi) là bao nhiêu giây?
  2. Đề xuất phương án giảm latency, demo + show thời gian đã giảm

Gợi ý các hướng tối ưu

HướngTiết kiệmTrade-off
Đổi model nhanh hơn (Sonnet → Haiku, GPT-4o → GPT-4o-mini) 2-5x Chất lượng có thể giảm
Streaming response — trả từng token thay vì chờ xong Cảm nhận nhanh hơn Cần frontend support
Cache câu hỏi đã hỏi (Registry-level hoặc agent-level) ~100% cho cache hit Stale data nguy hiểm cho luật
Connection pooling cho httpx ~100ms / call —
Bỏ analyze_law nếu không thật cần (flatten graph) ~1 LLM call Mất context cho specialists
Giảm system prompt (token càng ít, latency càng nhanh) ~10-20% Có thể giảm độ chính xác
Parallel hơn ở aggregate — gọi LLM tổng hợp luôn khi 1 specialist xong, đừng chờ cả 3 ~30-50% Cần redesign graph (streaming aggregate)

11. Câu hỏi tự kiểm tra

  1. Ưu điểm của A2A protocol so với gRPC hoặc REST thông thường?
  2. Tại sao cần Registry service — sao không hardcode URL trong code?
  3. Làm thế nào để prevent infinite delegation loops?
  4. Nếu Compliance Agent crash giữa request, hệ thống nên phản hồi như thế nào?
  5. Khi nào nên ở lại Stage 4 (in-process), khi nào nên lên Stage 5 (distributed)?
  6. Trace propagation có thể thay thế bằng cái gì khác (OpenTelemetry, Jaeger)?
