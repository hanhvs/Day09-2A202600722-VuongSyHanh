Stage 2 — LLM + RAG & Tools

Grounding LLM bằng knowledge base và function calling thủ công

⭐⭐ Trung bình 30 phút LangChain · @tool · bind_tools

1. Ý tưởng

Stage 1 chỉ dựa vào training data → hay bịa. Stage 2 cho LLM 2 khả năng mới:

  • RAG (Retrieval-Augmented Generation) — tra cứu knowledge base trước khi trả lời
  • Tool use (Function calling) — gọi Python function để tính toán/truy vấn data

Tool-call loop (manual — viết tay)

1. Bind tools vào LLM
→
2. LLM chọn tool + args
→
3. Code execute tool
→
4. Đẩy kết quả về LLM
→
5. LLM tổng hợp câu trả lời
⚠️ Hạn chế: chỉ 1 vòng tool call. Nếu LLM cần search lại lần 2, ta phải tự code thêm vòng lặp → đến Stage 3 (ReAct) thì agent tự lặp.

3. Knowledge Base — Keyword RAG

Demo dùng keyword matching đơn giản (production sẽ dùng vector store như Pinecone/Weaviate/Qdrant).

LEGAL_KNOWLEDGE = [
    {
        "id": "ucc_breach",
        "keywords": ["breach", "contract", "remedies", "damages", "ucc"],
        "text": "Under the UCC Article 2, remedies include ..."
    },
    # ...
]

@tool
def search_legal_database(query: str) -> str:
    q = query.lower()
    scored = [(sum(1 for kw in e["keywords"] if kw in q), e)
              for e in LEGAL_KNOWLEDGE]
    scored = [s for s in scored if s[0] > 0]
    scored.sort(reverse=True)
    return "\n\n".join(f"[{e['id']}] {e['text']}" for _, e in scored[:2])
💡 Tip về keyword matching tiếng Việt: nếu keyword là cụm 2 chữ ("sa thải", "lao động") thì phải dùng substring match (kw in query), không dùng set intersection trên từ đơn. Lý do: split() sẽ tách "sa thải" thành ["sa", "thải"] và không match được.

4. Anatomy của @tool

LLM "đọc" 3 thứ để quyết định gọi tool:

  1. Tên function — phải mô tả đúng việc nó làm
  2. Docstring — đây là phần LLM dựa vào nhiều nhất
  3. Type hints — schema args (kiểu, default, optional...)

5. Bài tập đã hoàn thành

Exercise 2.1 — Thêm entry mới vào KB

File: exercises/exercise_2_1_knowledge.py

Thêm entry labor_law chứa Điều 36 Bộ luật Lao động VN 2019 (6 trường hợp đơn phương chấm dứt HĐLĐ + thời gian báo trước).

Kết quả thực nghiệm

Câu hỏi: "Theo luật lao động Việt Nam, doanh nghiệp có quyền sa thải nhân viên trong những trường hợp nào và phải báo trước bao lâu?"

LLM gọi search_legal_knowledge với query "luật lao động Việt Nam sa thải nhân viên..." → match entry labor_law → tổng hợp câu trả lời:

  • ✅ Liệt kê đủ 6 trường hợp đơn phương chấm dứt HĐLĐ
  • ✅ Cite đúng "Bộ luật Lao động 2019"
  • ✅ Đưa đủ 3 mốc báo trước: 45 / 30 / 3 ngày

Bug đã sửa trong lúc làm

Lần chạy đầu trả "Không tìm thấy" vì search_legal_knowledge dùng set(query.split()) & set(keywords) — không match được keyword 2 chữ tiếng Việt. Đã đổi sang substring match (any(kw.lower() in query.lower() for kw in keywords)).

Exercise 2.2 — Tạo tool mới

File: exercises/exercise_2_2_new_tool.py

Thêm tool thứ 2 check_statute_of_limitations(case_type) hỗ trợ 7 loại vụ án (contract, tort, property, nda, trade_secret, employment, criminal).

Kết quả thực nghiệm

Câu hỏi: "Tôi muốn kiện công ty cũ vì vi phạm NDA và đánh cắp trade secret. Cho tôi biết các remedies và thời hiệu khởi kiện."

LLM tự gọi cả 2 tools cùng vòng:

  • 🔧 search_legal_knowledge(query="remedies for NDA violation and trade secret theft") → trả về remedies từ UCC
  • 🔧 check_statute_of_limitations(case_type="nda") → trả về "4 năm hoặc 3 năm dưới DTSA"

→ LLM tổng hợp thành câu trả lời có cả remedies và thời hiệu khởi kiện trong một response.

💡 Insight quan trọng: Khi câu hỏi gồm nhiều ý, LLM tự quyết định cần bao nhiêu tool calls — không phải code quyết định. Đây là tiền đề cho ReAct agent ở Stage 3.
